## 一、项目概述

### 1.1 项目背景

**易美传播**是一家专注于小红书 KOS（Key Opinion Sales）矩阵营销的代理公司，服务覆盖母婴、大健康、家居家装、汽车、酒类、医药、珠宝、食品等 7+ 个行业赛道，客户包括飞鹤、蒙牛、领克、极氪、林氏家居、董酒、老庙等。

当前售前提案流程依赖人工经验，存在以下痛点：
- 行业洞察和竞品分析依赖人工搜集，耗时且不全面
- 历史提案和案例分散在飞书文件夹中，检索效率低
- 提案撰写标准化程度不足，质量依赖个人能力
- 缺乏系统化的知识沉淀和复盘机制

### 1.2 项目目标

搭建一个**解决方案专家 Agent**，面向内部售前团队，通过 9 个可插拔 Skill 编排自动化提案生成流程，将售前提案效率提升 3-5 倍，同时保证提案质量的一致性和可追溯性。

### 1.3 成功标准

| 指标 | 目标值 |
|------|--------|
| 提案生成时间 | 从 2-3 天缩短至 4 小时内 |
| 行业数据覆盖 | 实时采集小红书行业/竞品/客户数据 |
| 知识复用率 | 历史提案和案例的检索命中率 > 85% |
| 提案规范性 | 100% 遵循标准提案骨架结构 |
| 格式输出 | 支持飞书 Slides + Docx + PPTX 导出 |

## 三、工作流设计

### 3.1 整体流程（6 阶段 + 2 决策点 + 1 反馈循环）

```
阶段0: 项目判断 ──→ 阶段1: 需求整理（双轨并行）──→ 阶段2: 方案设计
                                                      │
                                                      ▼
阶段5: 审核提交 ←── 阶段4: 视觉优化 ←── 阶段3: 内容撰写
    │                  (循环修改)
    ▼
  宣讲 → 复盘归档
```

### 3.2 阶段详细流程

#### 阶段 0：项目判断

| 步骤 | Skill | 输入 | 输出 | 依赖物料库 |
|------|-------|------|------|-----------|
| 0.1 | S1 商机评估 | 客户初步需求、行业、预算 | 评估结论（跟进/放弃）+ 项目分类 | 品牌信息表、提案复盘库 |

**决策节点 D1：** 商机评估 → 不跟进则终止并归档；跟进则进入阶段 1。

#### 阶段 1：需求整理（双轨并行）

| 轨道 | 步骤 | Skill | 输入 | 输出 | 依赖物料库 |
|------|------|-------|------|------|-----------|
| A（行业专家线） | 1.1 | S3 行业洞察 | 行业、品类、竞品名单、客户品牌名 | 现状分析报告 → 洞察分析报告 | 行业策略库、竞品分析库、增长模型库、小红书实时数据 |
| B（方案专家线） | 1.2 | S2 需求诊断 | 客户沟通记录、招标文件 | 需求文档 + 品牌资源提取 | 品牌信息表、提案模板库 |

**汇合点：** 轨道 A + 轨道 B 输出 → S4 客户洞察 → 客户洞察汇总文档。

#### 阶段 2：方案设计

| 步骤 | Skill | 输入 | 输出 |
|------|-------|------|------|
| 2.1 | S5 方案设计 | 客户洞察汇总 + 洞察分析报告 + 需求文档 + 标准提案框架 | 品牌增长全案方案 |

**品牌增长全案方案结构：**
- 运营策略 + 执行方案（行业洞察 / 客户诊断 / 竞品对标 / 增长策略 / 平台策略 / 执行方案）
- 工具赋能 + 预算规划（产品能力匹配 / 技术架构 / 预算规划）
- 案例展示
- 实施路径 & 交付物

#### 阶段 3：内容撰写

| 步骤 | Skill | 输入 | 输出 |
|------|-------|------|------|
| 3.1 | S6 案例匹配 | 行业 + 客户类型 + 痛点 + 方案模块 | 匹配案例卡片 |
| 3.2 | S7 内容生成 | 品牌增长全案方案 + 案例卡片 | 方案PPT初稿 |

#### 阶段 4：视觉优化

| 步骤 | Skill | 输入 | 输出 |
|------|-------|------|------|
| 4.1 | S8 格式输出 | 方案PPT初稿 | 预演PPT（飞书 Slides）+ 详细方案（飞书 Docx）+ PPTX 导出 |

#### 阶段 5：审核提交

| 步骤 | 角色 | 输入 | 输出 |
|------|------|------|------|
| 5.1 | 人工审核 | 预演PPT + 详细方案 | 通过/驳回 |
| 5.2 | 提案宣讲 | 最终版提案 | 宣讲材料 |
| 5.3 | S9 复盘归档 | 提案终稿 + 竞标结果 | 归档资料 + 物料库更新 + 复盘报告 |

**决策节点 D2（内部评审）：** 不通过 → 返回阶段 3/4 修改；通过 → 宣讲。

## 四、Skill 详细规格

### 4.1 Skill 1：商机评估

| 属性 | 内容 |
|------|------|
| 对应角色 | 销售 |
| 对应阶段 | 阶段 0 - 项目判断 |
| 输入 | `{ client_name, industry, budget_range, initial_requirements }` |
| 输出 | `{ go_or_no_go, project_type, confidence_score, risk_factors, recommendation }` |
| 核心逻辑 | 1. 检索品牌信息表判断是否老客户 2. 检索提案复盘库获取同类成败经验 3. 基于项目分类模型判断竞标类型 4. 综合评分输出建议 |
| 依赖物料库 | 品牌信息表、提案复盘库 |

### 4.2 Skill 2：需求诊断

| 属性 | 内容 |
|------|------|
| 对应角色 | 销售 + 方案专家 |
| 对应阶段 | 阶段 1 - 需求整理（轨道 B） |
| 输入 | `{ communication_records, rfp_document (可选) }` |
| 输出 | `{ requirement_document: { client, industry, pain_points, goals, budget, timeline, scope }, brand_assets: { ppt_template, design_guidelines, logo } }` |
| 核心逻辑 | 1. 填充招标拆解表 2. 客户需求翻译到标准提案骨架 3. 信息缺口检测 → 主动追问 4. 品牌资源提取 |
| 依赖物料库 | 品牌信息表、提案模板库 |

### 4.3 Skill 3：行业洞察

| 属性 | 内容 |
|------|------|
| 对应角色 | 行业专家 |
| 对应阶段 | 阶段 1 - 需求整理（轨道 A） |
| 输入 | `{ industry, category, competitors: [name, ...], client_name }` |
| 输出 | `{ industry_analysis: { trend_keywords, volume_trend, content_distribution, hot_authors }, competitor_analysis: [{ name, kos_scale, monthly_notes, avg_interaction, content_strategy, strength, weakness }], client_diagnosis: { account_health_score, content_quality_score, sentiment_ratio, key_issues, improvement_areas }, growth_analysis: { platform_strategy, growth_model } }` |
| 核心逻辑 | 1. 小红书数据采集（行业关键词 / 竞品品牌号 / 客户账号） 2. NLP 分析（情感分析 / 内容分类 / 声量趋势） 3. 历史数据补充（行业策略库 / 竞品分析库 / 增长模型库） 4. 生成结构化分析报告 |
| 依赖物料库 | 行业策略库、竞品分析库、增长模型库 |
| 外部数据源 | 小红书公开数据 |

**小红书数据采集维度：**

| 维度 | 采集内容 | 分析产出 |
|------|----------|----------|
| 行业 | 品类热搜词、趋势笔记、头部达人、用户讨论热点 | 声量趋势图、内容热词云、品类增长机会 |
| 竞品 | 品牌号笔记、KOS 矩阵、评论区互动、投放素材 | 竞品策略拆解、KOS 矩阵对标、差异化机会 |
| 客户 | 现有账号、笔记内容、评论区舆情、粉丝画像 | 账号健康度评分、内容质量评分、舆情预警 |

### 4.4 Skill 4：客户洞察

## 五、8 大物料库设计

| 物料库 | 数据来源 | 存储结构 | 向量化 | 更新方式 |
|--------|----------|----------|--------|----------|
| 品牌信息表 | 各客户提案中的品牌背景页 | 结构化 JSON | 否 | 提案后自动 + 手动 |
| 行业策略库 | 行业通案中的洞察和方法论 | 分段向量 | 是 | 提案后自动沉淀 |
| 竞品分析库 | 提案中的竞品对标 + 小红书实时采集 | 分段向量 | 是 | 实时采集 + 提案后沉淀 |
| 增长模型库 | 方法论文档中的增长策略 | 分段向量 | 是 | 手动维护 + 提案后优化 |
| 产品解决方案库 | 产品介绍、培训资料、报价单 | 分段向量 | 是 | 手动维护 |
| 案例标签库 | 历史案例文件 | 结构化 JSON + 向量 | 是 | 提案后自动沉淀 |
| 提案模板库 | PPT 模板、公司介绍、标准框架 | 文件存储 | 否 | 手动维护 |
| 提案复盘库 | 提案结束后自动生成 | 结构化 JSON | 是 | 每次提案后自动 |

## 六、知识闭环机制

```
提案开始 ──→ 调用 8 大物料库 ──→ 提案执行
                  ↑                    │
                  │                    ↓
                  └──── S9 复盘归档 ──── 提取 + 更新
```

每个提案生命周期 = 一次物料库的"消费 + 反哺"，确保知识库随业务增长持续进化。

## 七、关键边界与约束

### 7.1 明确包含

- 基于飞书文件夹中 30+ 份历史提案和案例构建初始知识库
- 小红书公开数据采集（行业/竞品/客户三维度）
- 飞书生态内 Slides + Docx 自动生成 + PPTX 导出
- 人工审核节点（商机评估 + 内部评审）
- 复盘归档与物料库自动更新

### 7.2 明确不包含

- 小红书数据采集的合规性审核（需用户自行确认合规）
- 客户关系管理（CRM）集成
- 合同管理和报价审批流程
- 飞书文档的实时协作编辑
- 多语言提案生成（仅支持中文）

### 7.3 技术约束

- 数据采集频率需控制在小红书反爬策略允许范围内
- 飞书 Slides API 不支持直接从外部 URL 插入图片，需先上传到飞书 Drive
- 各 Skill 间通过结构化 JSON 传递数据，不共享内存状态
- 人工审核节点为阻塞式，需等待用户反馈后才能继续

### 8.2 服务组件清单

| 服务 | 技术栈 | 端口 | 说明 |
|------|--------|------|------|
| orchestrator | Python / FastAPI + LangGraph | 8000 | 核心编排引擎，调度 9 个 Skill |
| web-chat | React / Next.js | 3000 | 内部售前团队 Web 界面 |
| feishu-bot | Python / FastAPI + lark-cli | 8080 | 飞书消息 Bot 接入 |
| worker | Python + Celery | — | 小红书数据采集 + 复盘归档 |
| scheduler | APScheduler / Celery Beat | — | 物料库增量更新、数据清理 |
| qdrant | Qdrant 向量数据库 | 6333 | 8 大物料库的向量检索 |
| postgres | PostgreSQL 15 | 5432 | 品牌信息表、案例标签等结构化数据 |
| redis | Redis 7 | 6379 | Skill 间消息队列 + 热点缓存 |
| ollama | Ollama + qwen2.5 | 11434 | 本地 LLM 推理（可选） |
| nginx | Nginx | 80 | 反向代理 + 静态资源 |

### 8.3 LLM 方案：混合模式

| 任务类型 | 使用模型 | 说明 |
|----------|----------|------|
| 轻量任务（需求诊断、案例匹配、复盘归档） | 本地 Ollama + qwen2.5:7b | 响应快、零成本 |
| 重量任务（方案设计、内容生成、行业洞察分析） | 云端 API（Claude / 通义千问） | 质量有保障 |
| 路由逻辑 | 编排 Agent 根据任务复杂度自动选择 | 可配置切换阈值 |
